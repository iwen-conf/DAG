# 01 · 总体架构

## 1.0 交付边界（D-19，先划清楚）

**本系统交付两个二进制，且不是 Agent：**

| 交付物 | 是什么 | 不是什么 |
|--------|--------|----------|
| `sunmao-server` | 提供 API 的服务：图/任务/租约/事件/产物的唯一管理者，**一服务多项目** | 不内嵌 LLM，不调模型，不执行提示词 |
| `sunmao` | 人用客户端：TUI 配置 base_url、查看项目、目录绑定、范围操作 | 不是 Worker，不替 Agent 干活 |

Planner / Worker（Claude Code 等 LLM 会话）是**纯外部 API 调用方**——和本系统的关系等于 `kubectl` 之于 Kubernetes 里的应用：系统只管任务板和产物，谁来领任务、怎么完成，系统不知道也不关心。仓库里的提示词模板（M5）是**使用文档**，不是系统组件。

## 1.1 系统形态：一个服务 + 两类客户端

```text
┌─────────────────────────────────────────────────────────┐
│                     sunmao-server (单进程)                   │
│                                                          │
│  HTTP API ◄──── Worker 端点（claim/heartbeat/complete…） │
│     │      ◄──── Planner 端点（expand/publish…）          │
│     │      ◄──── 人类端点（scope/approve-major/UI 数据）   │
│     │      ◄──── SSE /events（实时推送，源自 PG NOTIFY）   │
│     │                                                    │
│  ┌──────────┐  ┌──────────┐  ┌───────────┐              │
│  │ 图引擎    │  │ 调度器    │  │ 租约回收器 │ ← tokio 定时  │
│  │ (校验/    │  │ (claim-  │  │ (过期扫描) │              │
│  │  派生)    │  │  next)   │  └───────────┘              │
│  └──────────┘  └──────────┘                              │
│        │              │                                  │
│  ┌─────┴──────────────┴─────┐   ┌────────────────────┐  │
│  │       PostgreSQL          │   │  git 子进程         │  │
│  │  (project/node/event/     │   │  (diff 核验/commit) │  │
│  │   artifact + NOTIFY)      │   │  作用于各项目仓库    │  │
│  └───────────────────────────┘   └────────────────────┘  │
└─────────────────────────────────────────────────────────┘
         ▲                    ▲                   ▲
    Worker Agent          Planner Agent      人 (sunmao TUI / 命令)
   (外部 LLM 会话)       (外部 LLM 会话，D-12)
```

要点：

- **一服务多项目**（D-19）：项目 = 一个 git 仓库地址（`repo_path`，服务端唯一）。图、任务、事件、产物全部按 `project_id` 隔离；git 子进程按项目的 `repo_path` 分别操作。
- **sunmao-server 是唯一常驻进程**（FR-05.1：所有协作只经 Task Service）。Planner / Worker 都是**外部 Agent**，通过 HTTP 访问；系统内部没有内嵌 LLM（D-19）。
- **Planner 是角色不是服务**（D-12）：单 Planner = 「同一时刻至多一个规划会话在提交图变更」，由 server 端的图版本乐观锁保证，而非部署约束。
- **人走同一套 API**（D-14/D-18）：无认证，调用方以 `actor` 标识自报身份（`human` / `agent:<id>`），范围操作端点靠**暴露面约定**只给人用（见 03 §暴露面分层）。
- **Git 只在完工时被调用**（D-09=A）：server 对目标项目树执行 `git status/diff/add/commit`，运行中不碰 git。

## 1.2 Rust Workspace 划分

```text
sunmao/
├── Cargo.toml            # [workspace]
├── crates/
│   ├── sunmao-core/      # 纯逻辑，零 IO
│   ├── sunmao-store/     # PG 访问 + git 适配
│   ├── sunmao-server/    # HTTP + SSE + 后台任务，二进制
│   └── sunmao-cli/       # 人用 CLI，二进制名 `sunmao`（D-20）
└── docs/
```

| crate | 职责 | 关键约束 |
|-------|------|----------|
| **sunmao-core** | 领域类型（NodeId/State/WriteScope/semver）、状态机迁移表、图算法（环检测、写冲突判定、派生态计算、受影响子图）、编译校验（FR-03.3 全部规则） | **不依赖 tokio/sqlx/git**；全部纯函数，property test 主战场。D-01 全自动后校验是唯一质量门，独立成纯逻辑 crate 是测试完备性的前提 |
| **sunmao-store** | repository 层：node/event/artifact/attempt 的 PG 读写（sqlx）、事件+投影同事务写入（D-17）、claim-next SQL、NOTIFY 发送；`GitWorkspace`（diff 核验 / 按写范围 commit，子进程调用） | 所有 SQL 集中于此；事务边界在这层显式化 |
| **sunmao-server** | axum 路由、SSE 广播（LISTEN → 内存 fan-out）、租约回收器（tokio interval）、Drain 协调 | 只做编排：解析请求 → 调 core 校验 → 调 store 落库 |
| **sunmao-cli** | 人类操作入口：**TUI**（ratatui：配置 base_url、项目列表/详情、任务树浏览）+ 子命令（`init/tree/task/scope/approve`），打 server 的 HTTP API；二进制名 `sunmao` | 目录绑定解析默认项目（见 §1.5） |
| **Web UI** | 人用控制台，由 `sunmao-server` 静态托管于 `/ui/`（项目列表、任务树、事件、scope/approve/rebuild）；**零 LLM**，只调同源 `/v1` API | D-19 人类面；与 CLI 并存 |

依赖方向：`sunmao-cli → (HTTP) → sunmao-server → sunmao-store → sunmao-core`；core 被所有层引用，反向禁止。

## 1.3 技术选型

| 关注点 | 选择 | 理由 |
|--------|------|------|
| HTTP | **axum** 0.8 | tokio 生态一等公民，SSE 原生支持 |
| PG 访问 | **sqlx**（运行时查询 + 编译期检查混用） | 无 ORM 抽象税；`query_as!` 编译期校验列类型；迁移用 `sqlx migrate` |
| Git | **子进程调用 `git` CLI** | D-09=A 只需要 status/diff/add/commit 四个动作；libgit2/gitoxide 引入依赖但覆盖不了 CLI 的行为兼容性（hooks、config）；子进程输出用 `--porcelain` 稳定解析 |
| 实时推送 | **PG LISTEN/NOTIFY → SSE** | D-17 事件插入即 NOTIFY；SSE 比 WebSocket 简单且单向足够（客户端命令走普通 HTTP） |
| TUI | **ratatui + crossterm** | D-19 的 CLI 交互模型；表单（base_url 设置）+ 列表（项目/任务树）两类界面，无重前端 |
| ID | **前缀 ULID**（`nd_`/`ev_`/`at_`/`ar_`） | 时间有序（分页友好）、字符串可读、无中心分配 |
| 时间 | 全部 `timestamptz`，server 不存本地时区 | 租约比较在 PG 内完成（`now()`），避免时钟漂移 |
| 序列化 | serde + JSON；JSONB 存半结构字段 | Task 契约/attempts payload/事件 payload 免建表（D-09 理由） |

**明确不引入**：消息队列（PG 即队列，SKIP LOCKED）、缓存（派生态实时算，D-16）、ORM、认证中间件（D-18）、WebSocket、多进程 worker 池（Worker 在系统外部）。

## 1.4 关键架构决定（本层新增，ADR 式）

### A-01 调度器是"查询"不是"进程"

没有常驻调度循环。`claim-next` 端点内的一条 SQL（`FOR UPDATE SKIP LOCKED`）就是调度器（D-05）；Ready 的判定条件（依赖满足、祖先允许、能力匹配、写范围无占用）全部编码为该查询的 WHERE 子句与预计算列。唯一的后台任务是**租约回收器**（interval 扫描过期租约）。

推论：系统崩溃恢复 = PG 重连，无内存状态需要重建（14.1 可恢复性）。

### A-02 写冲突的三道防线及其分工

| 时机 | 机制 | 决策 |
|------|------|------|
| 规划期 | 编译校验拒绝兄弟 Ready 任务写范围相交（前缀判定） | FR-03.3 |
| 认领期 | claim-next 排除「写范围与任何 Running 任务相交」的候选 | D-10 引申——静态检查只覆盖同批发布，跨批次并行靠这里 |
| 完工期 | git diff 越界 → 判失败 | D-10 |

认领期防线是需求文档未明说但必须补的：两个不同批次发布的任务可能写范围相交且同时 Ready，只靠静态+事后不够。

### A-03 事件是唯一写入口

store 层公开的所有变更方法签名形如 `apply(event) → 同事务(INSERT event + UPDATE 投影 + NOTIFY)`。**不存在绕过事件直改投影的代码路径**（D-17 的工程强化）。投影重建工具（`sunmao admin rebuild`）从事件流全量重放，用于灾后校验与 schema 演进。

### A-04 派生态的计算位置

Blocked/Ready 派生（D-16）、Package 聚合进度（Running/Completed）都在 **SQL 视图/递归 CTE** 内完成，sunmao-core 提供同构的纯函数版本用于校验与测试（双实现，property test 断言二者一致）。UI 读视图，调度读预计算列（见 02 §索引策略——`ready` 是物化列，由触达它的事务同步维护，避免每次 claim 全图递归）。

## 1.5 部署与 CLI 使用形态（D-19）

```bash
# ── 服务端（一次，常驻）───────────────────────────
sunmao-server --db postgres://localhost/sunmao        # 监听 127.0.0.1:7420

# ── CLI 首次配置（TUI）──────────────────────────
sunmao config                # TUI：设置/修改 base_url → ~/.config/sunmao/config.toml
sunmao projects              # TUI：项目列表 / 详情（也可 --json 走脚本）

# ── 在项目目录里注册与绑定 ────────────────────────
cd /path/to/novel-site
sunmao init                  # 1. 取 git root 与 repo_path
                          # 2. 服务端创建/关联项目（repo_path 唯一）
                          # 3. 写仓库根 .sunmao.toml（project_id + base_url）
                          # 此后该目录及子目录内，一切 sunmao 命令默认指向本项目

# ── 日常（目录内，无需指定项目）───────────────────
sunmao tree                  # 任务树（TUI 可浏览/下钻）
sunmao scope close server.payment --reason "业务暂缓"
sunmao approve-major ar_xxx
```

**默认项目解析顺序**（CLI 内实现，服务端无感知）：

```text
--project <id> 显式参数
  > 仓库根 .sunmao.toml 的本地绑定
    > cwd 的 git root 反查服务端 project.repo_path
      > TUI 项目选择器（交互式兜底）
```

配置两层：全局 `~/.config/sunmao/config.toml`（base_url）；项目 `.sunmao.toml`（project_id，入 git 可选——单人场景无冲突）。

单二进制 + 一个连接串；`127.0.0.1` 硬编码为默认监听（D-18 的技术表达），改绑 `0.0.0.0` 需显式 flag `--unsafe-expose`。
