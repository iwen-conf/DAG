# 08 · 边界拆分方法论

> 来源：完整对话第 1 轮（如何拆分任务边界 / 上下文）  
> 用途：Planner 拆 Package / Context 时的**业务边界准则**；与运行时调度协议正交。

## 8.1 核心原则

> **按照「变化原因（Reason to Change）」去拆，而不是按照数据 / 表 / 接口去拆。**

按库表拆（如巨型 `UserService`）最终会拆烂；稳定边界应对齐「为什么会改」「谁会改」「是否必须同事务」。

## 8.2 七层拆法

### 第一层：先找业务动作（Task）

不要先看数据库。先问：**用户到底在干什么？**

示例（小说网站）：

```text
登录 / 注册 / 阅读小说 / 搜索 / 收藏 / 评论
购买会员 / 充值 / 上传小说 / 审核小说 / 推荐小说
```

这些是业务 Task，不是表名。

### 第二层：任务里的业务规则 → Context

「注册」内含：手机合法、验证码、用户名重复、密码规则、创建用户、欢迎消息、送积分/优惠券……

这些整体属于 **Register Context**，不是笼统的 User Context。

反模式：

```text
UserService.Create / Update / Delete / Login / Avatar / Permission / OAuth ...
```

正模式：拆成多个 Context，例如：

```text
Register | Login | Profile | Security
```

### 第三层：哪些数据总是一起变化（聚合）

小说：标题、简介、作者、分类、封面 → 常一起改 → 可同聚合。

阅读量、点赞、收藏 → 变化频率完全不同 → 拆成 `Novel` 与 `NovelStatistic`。

### 第四层：生命周期

订单：创建 → 支付 → 退款 → 取消 → 完成 → 同一 Context。  
优惠券：领取 → 使用 → 过期 → 生命周期不同，不要塞进订单。

### 第五层：必须同事务 vs 最终一致

| 必须同事务 | 可拆出最终一致 |
|------------|----------------|
| 创建订单 + 扣库存 | 发短信 / 发 MQ / 加积分 |

### 第六层：Actor（参与者）

| Actor | 示例动作 |
|-------|----------|
| 后台 | 审核小说 |
| 作者 | 上传小说 |
| 用户 | 阅读 |
| AI | 生成简介 |

Actor 不同，天然可能是不同 Context。

### 第七层：Bounded Context 目录（示例）

```text
Identity     Login / Register / Permission / OAuth
User         Profile / Avatar / Preference
Novel        Novel / Chapter / Category / Tag / Draft
Reading      History / Bookmark / Progress / ReadingRecord
Comment      Comment / Reply / Like
Search       Index / Recommend / Suggest
Payment      Order / Membership / Wallet / Coupon
Statistic    PV / UV / HotRank / Report
Notification SMS / Email / Push / Inbox
```

每一个是 **Context**，不是一个 Service 类。

## 8.3 五问法（日常判断）

对任一功能连续问：

```text
① 谁操作？
② 为什么操作？
③ 什么情况下修改？
④ 和谁必须一起成功？
⑤ 将来谁最可能修改它？
```

答案若**完全不同** → 应该拆。

### 示例：用户头像

| 问 | 答 |
|----|----|
| 谁 | 用户自己 |
| 为什么 | 改资料 |
| 何时 | 偶尔 |
| 事务 | 不必与登录一起 |
| 未来 | 可能接 OSS |

→ **Profile Context**

### 示例：JWT

| 问 | 答 |
|----|----|
| 谁 | 登录系统 |
| 为什么 | 登录成功 |
| 事务 | 必须与 Session 一致 |
| 未来 | OAuth |

→ **Identity Context**（不是 User）

## 8.4 坏味道：Context 划错

若出现 `UserService` 8000 行且塞入 Login / OAuth / Avatar / Permission / Statistics… 几乎可确定 **Context 已划错**。

好设计：一个 Context 只围绕**一种能力（Capability）**，有独立业务语言、变化节奏与生命周期。

## 8.5 与 Agent 的对应关系

对话明确类比：

> Agent 不应按 Tool 划分，而应按 **Task Context** 划分。

「检索知识 / 写文档 / 调外部服务 / 执行代码」是不同能力边界，不要把所有工具堆进一个 Agent。

映射到本系统：

| 方法论对象 | 系统对象 |
|------------|----------|
| Bounded Context | 高层 Package（如 Identity、Novel） |
| 业务 Task | Feature 或叶子 Task |
| 变化原因不同 | 不要共用同一写权限 / 同一模块节点 |
