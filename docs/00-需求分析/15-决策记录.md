## D-01 · Planner 草案是否自动发布 ✅ 已确认

- **决策**（用户拍板 2026-07-16）：**全自动**——Planner 的展开/拆分草案通过编译校验（FR-03.3：环、写冲突、悬空依赖等）后直接发布生效，不设人工审批门。
- **人的位置**（FR-01.4 重释）：从「事前审批门」改为「事后干预权」——人随时可在 UI 编辑、删除、重拆任意子图；安全网 = 范围控制（Pause/Close，FR-09）+ 全量审计（D-17 事件表）+ 失败链路暴露（D-06/D-07）。
- **原建议存档**：整理者原建议人工批准（防早期 Planner 拆错浪费 Worker 执行），被用户否决。保留 `manual_approve` 配置开关作为日后可选项（默认关闭）。
- **影响**：Package `Planning → Ready` 自动流转，状态机不存在「待批准」环节；编译校验成为发布前唯一强制关卡，其完备性（FR-03.3/03.4）的优先级相应提高。

## D-02 · 节点类型集合 ✅ 已确认（授权落定）

- **决策**：运行时仅 `package | task` 两类；Project/Module/Feature 用可选 `layer` 标签表达，只影响展示不进调度。
- **理由**：协议不变量最少（`type==task → children==[]`，[07](./07-节点协议.md) §7.3）；调度代码只依赖二分；标签可后加、不破坏兼容。
- **影响**：FR-02 与节点协议维持现状；UI 分层视图读 `layer`；14.2 D-02 落定。

## D-03 · Ignore 的建模 ✅ 已确认（授权落定）

- **决策**：不设独立状态；用 `Closed + permanent: true` 表达「永不做」。
- **理由**：Closed 与 Ignore 对调度和 Planner 的行为效果完全相同（不派发、不展开），差异只是「是否预期恢复」——这是属性不是状态。少一个状态 = 少一行迁移矩阵与权限矩阵。
- **影响**：[04-状态机](./04-状态机.md) 的「建议增加 Ignore」按此收敛；Package 扩展字段加 `permanent`；FR-09.7 落定。

## D-04 · Pause/Close 时正在 Running 的 Task ✅ 已确认

- **决策**：默认 **Drain**——已 Claim/Running 的任务允许跑完并走完 Validate，只停止派发新任务。提供显式 `--force`（终止租约、回收或标记 Cancelled）作运维兜底。
- **理由**：强杀产生半成品产物，违背「不出现 Done 但产物不可读」（14.1 可恢复性 3）；Drain 与 K8s cordon/drain 心智一致。
- **影响**：Closed 的生效语义 = 「不再进入 next」而非「子树立即静止」；UI 需显示 draining 中的任务数。

## D-05 · Claim 租约 ✅ 已确认（含用户补充的接力协议）

- **决策**：必须有。Claim 即建租约（`lease_expires_at`），Worker 心跳续租；过期由回收器原子回收至 Ready 并递增 attempt。**Claimed 与 Running 两态都保留**（领了未开工 / 已开工），租约自 Claimed 起算。
- **接力协议**（用户拍板 2026-07-16）：租约过期后任务**允许交给其他 Agent 接手**，但接力不是重新开始，必须满足：
  1. **旧执行立即作废**：原租约持有者被标记为失效；若它此后仍提交结果，一律拒收（防幽灵写入——旧 Agent 可能只是挂起而非死亡）；
  2. **现场审查**：新 Agent 领取时，系统在任务输入中注入交接上下文——上一手的 attempt 记录、失败/中断原因、已产生的在制品（项目树中该任务写范围内的未 commit 变更）、当前项目与 Task 状态；新 Agent 必须先审查现场（在制品可复用 / 需清理重做）再决定继续方式；
  3. **如实上报，不得隐瞒**：新 Agent 必须把现场审查结论回写系统（在制品状态、上一手遗留问题、自己的处置决定），进入 D-17 事件表与 `attempts[]`，作为审计与后续交接的输入；
  4. **进度同步**：接手前强制同步一次数据/进度快照（拉取最新图状态、依赖产物版本、该任务写范围现状），确保新 Agent 基于当前真实状态工作，而非过期认知。
- **理由**：Worker 是外部 AI 进程，崩溃、挂起、断网不可避免；租约是唯一不依赖 Worker 配合的回收机制（14.1 可恢复性 1/2）。接力协议解决「回收后的任务不是白纸」这一现实——在制品有价值也有毒性，必须显式审查而非默默覆盖。注意：租约与认���无关（D-18 取消的是 auth token），它是崩溃恢复机制，单人使用也必须有。
- **影响**：
  - `attempts[]` 每条含交接审查报告字段（供 D-06 重试注入上下文复用同一结构）；
  - 领取接口对「接力任务」（attempt > 1 或有在制品）返回交接上下文包；
  - fencing：提交接口校验提交者 = 当前租约持有者且租约有效；
  - 过期回收计入 attempt 上限（D-06），防挂起 Agent 无限接力。
- **实现**（PostgreSQL，D-09 已确认）：

```sql
-- 原子 claim-next：条件匹配 + 认领一条语句完成，天然防抢
WITH next AS (
  SELECT id FROM task
  WHERE state = 'ready' AND required_capabilities <@ :agent_caps
  ORDER BY priority DESC, id
  FOR UPDATE SKIP LOCKED
  LIMIT 1
)
UPDATE task t SET owner = :agent_id, state = 'claimed',
       lease_expires_at = now() + :ttl,
       lease_token = gen_random_uuid()          -- fencing token：旧持有者提交即拒
FROM next WHERE t.id = next.id
RETURNING t.*;

-- 租约回收器（定时执行）：回收 = 记事件 + 递增 attempt + 作废旧 token
UPDATE task SET state = 'ready', owner = NULL, lease_token = NULL
WHERE state IN ('claimed','running') AND lease_expires_at < now();
```
- **影响**：[06](./06-调度与任务服务.md) 最小 API 需补 `heartbeat / complete / fail` 端点；为何不用 Temporal 承担此职责见附录 A。

## D-06 · 失败重试 ✅ 已确认

- **决策**：同一 Task 自动重试最多 2 次（attempt ≤ 3），把上次 Validator 失败输出注入下次输入；仍失败 → Failed 停住等人或触发 D-07 升级。**租约过期回收产生的接力（D-05）同计入 attempt 上限**，防挂起 Agent 无限接力。
- **理由**：AI 任务失败多为系统性原因（契约错、任务过大、输入产物有毛病），盲目重试不收敛只烧钱；带失败上下文的重试收益最高。
- **影响**：Task 增加 `attempts[]`（每次的 owner / 时长 / 失败或中断原因 / 交接审查报告——结构与 D-05 接力协议共用）；满足 FR-08.4 与 14.1 幂等要求。

## D-07 · Validator 失败后的返工形态 ✅ 已确认

- **决策**：**原 Task 重开**（回 Ready、attempt+1），不新建修复 Task；连续失败达上限、或失败被判定为拆分/契约问题时，才触发 Replanner 改受影响子图。
- **理由**：按失败新建 Task 会让图随失败膨胀、血缘断链；「同一节点多次尝试」与构建系统重试语义一致，历史进 `attempts[]` 不丢。
- **影响**：FR-08.5 收敛为两级策略；Replanner 触发条件成为架构设计的明确输入。

## D-08 · Contract 冻结与破坏性变更 ✅ 已确认

- **决策**（用户拍板 2026-07-16，选 **semver 完整版**）：Contract 是版本化 Artifact，采用 **semver 分级**：
  - **patch**（注释/文档/格式，不改语义）：Planner/Worker 可发布，不触发影响标记；
  - **minor**（向后兼容新增：新接口、新可选字段）：自动发布，仅通知性标记后继节点（不强制返工）；
  - **major**（破坏性：改签名、删字段、改语义）：**仅人可发布**，发布时系统自动计算并标记全部受影响后继节点（FR-10.4），交 Replanner 生成返工子图。
- **分级判定**：优先程序判定（schema diff 可机判新增 vs 删改），机判不确定时保守升级为 major；判级结果与依据进 D-17 事件表。
- **理由**：Contract 是并行安全的地基（00 §4.3），破坏性变更与 Close Package 同级属范围决策，理应人控；分级避免「改个注释也全量标记返工」的误伤。
- **影响**：FR-10.5 落定；Contract Artifact 携带 `version`（semver）字段，PG 记录版本链；D-01 全自动展开遇到依赖 major 变更未确认的子图时暂停该子图展开。

## D-09 · Artifact Store 选型 ✅ 已确认

- **决策**（用户拍板 2026-07-16，分两次）：
  - **元数据库 = PostgreSQL**（图、状态、租约、事件）。理由：环境已有 PG，零部署成本；`FOR UPDATE SKIP LOCKED` 是任务队列教科书原语（见 D-05）；`LISTEN/NOTIFY` 让 Worker 等待从轮询升级为推送；`WITH RECURSIVE` 直接支撑祖先/后代/受影响子图查询（FR-02.4）。SQLite 备选论证见附录 B。
  - **产物层 = 方案 A（轻量 Git）**：目标项目本身是 git 仓库；系统对 Git 的使用仅两点——①完工 `git diff` 核验越权写；②每任务完成一次 commit 留回溯。**无分支、无 worktree、无 merge**；并行安全靠独占写范围（写范围不相交则物理上无冲突）。
- **落定 FR**：FR-07.5（产物后端）→ 文件在项目树 + PG 记 `路径集合 + commit hash`；FR-11.4（隔离方案）→ 不引入工作树/分支/沙箱。
- **未选方案存档**：B 无 Git（PG 存哈希 + 快照自建回溯——放弃 diff 免费能力）；C 完整 Git 底座（worktree 隔离 + 分支 + merge 收口——单人 v1 偏重，大量 Agent 高并发写同区时再评估）。
- **历史勘误**：本文曾把「Git 存产物 + 四职责」标为已确认——实际当时用户仅确认了 PostgreSQL。Git 四职责原是整理者建议（原始需求 FR-07.5/FR-11.4 中 Git 仅为候选，对话原文仅两处涉及 git：层级类比；「三方合并都经常冲突」）。产物层方案 A 现已由用户明确选定。
- **影响**：「Done 但产物不可读」由「commit 成功才置 Done」保证；FR-13.3 还原现场 = PG 事件 + commit hash 双向引用；14.1 性能 1 的 Ready 查询用 PG 部分索引（`WHERE state='ready'`）实现。

## D-10 · 写集合检测（随 D-09=A 落定）

- **决策**：**声明式写范围**（Task 契约中的 glob/目录）+ **完工时 `git diff` 核验**：Worker 提交完成时，系统对项目树执行 diff，触及声明外路径 → 判失败（FR-06.5）。不做文件锁与实时拦截。
- **并发口径**：并行任务的写范围在规划期已保证互不相交（FR-03.3 静态检查），因此多任务在制品共存于同一项目树时，`git diff` 按「路径 ∈ 该任务写范围」归属变更，完工按任务写范围逐一 commit。
- **理由**：事前锁/沙箱成本高且约束 Worker 形态；事后按实际变更判定，与规划期静态检查构成事前+事后两层。
- **影响**：越权写成为标准 Validator（FR-08.2 已列）。

## D-11 · Merge 策略（随 D-09=A 落定）

- **决策**：**v1 没有 merge 这个环节**。独占写权限是唯一且充分的防线（[09](./09-小任务与并行条件.md) §9.4「并行的本质是无共享写」）：写范围不相交 → 不存在合并问题。无分支、无三方合并、无 AI Merge Agent。
- **Integration Task 的职责**改为**跨产物一致性验证**：如 Schema ↔ OpenAPI ↔ 代码是否一致（对话难点⑤），产出一致性报告；发现不一致 → 判失败并指名冲突产物，交 D-07 返工或 Replanner。
- **理由**：若「需要 merge」的情况频繁出现，说明写范围划分有问题，该修 Planner 的拆分，而不是引入 merge 机制。
- **影响**：FR-11.2/11.3 的「Merge/冲突报告」在 v1 具体化为「一致性验证/不一致报告」；FR-11.4 落定为「无隔离方案，独占写直写」。

## D-12 · 多 Planner ✅ 已确认（授权落定）

- **决策**：v1 单 Planner（图的单写者）；协议字段不为多 Planner 特化。
- **理由**：多 Planner 并发改图是全新的冲突治理问题域；单写者下图版本线性演进，审计与回滚简单。
- **影响**：FR-03 按单 Planner 实现；v2 可用「Planner 仅被授权展开特定子树」过渡到多 Planner。

## D-13 · 优先级与公平性 ✅ 已确认（授权落定）

- **决策**：只做整数 `priority`（默认 0），取任务按 `priority DESC, id ASC`；不做 deadline、成本、公平性。
- **理由**：一行排序换来手动插队能力，足够 v1；其余维度在没有真实负载数据前全是臆测。
- **影响**：[07](./07-节点协议.md) Task 扩展字段 `priority` 落定。

## D-14 · 人类能否直接 Claim/完成 Task ✅ 已确认（授权落定）

- **决策**：能。人以 capability 全集的特殊 Worker 身份走同一套 next / claim / complete 流程。
- **理由**：模型统一，不为人开旁门；也是 AI 反复失败时的逃生通道——人做完提交，图照常前进。D-18（无认证）下人和 Agent 本就同接口，该决策自然成立。
- **影响**：[05](./05-角色与协作.md) 权限矩阵不变（人 ⊇ AI）；UI 提供「认领给我」。

## D-15 · Worker 隔离（随 D-09=A 落定）

- **决策**：Worker **直接在项目树上工作**，不引入 worktree/分支/容器/远程执行。隔离 = 独占写范围（规划期保证互不相交）+ 完工 `git diff` 核验（D-10）兜底。
- **理由**：容器解决的是「不可信代码执行」，v1 的 Worker（Claude Code 等）已有自身沙箱；单人场景下独占写范围的成本远低于分支/工作树生命周期管理。
- **影响**：Drain（D-04）期间在制品直接留在项目树（未 commit）；`--force` 终止时用 `git checkout -- <该任务写范围>` 丢弃在制品。

## D-16 · Blocked：落库还是派生（新增）✅ 已确认（授权落定）

- **决策**：**派生视图**。真相源只有「祖先 Package 状态」与「depends 满足度」；`next` 查询时过滤，UI 需要时物化缓存。
- **理由**：落库则 Close/Reopen 需级联写整棵子树，且与调度端的祖先检查（07 §7.2）构成双真相源，必然漂移；派生零级联、天然一致。PG 递归 CTE（D-09）让祖先链检查成本可忽略。
- **影响**：[04-状态机](./04-状态机.md) 中 Blocked 标注为派生态；大子树查询用祖先路径索引解决（14.1 性能 1）。

## D-17 · 元数据存储结构：事件表 + 投影（新增）✅ 已确认（授权落定）

- **决策**：核心库采用 **append-only 事件表**（谁 / 何时 / 对哪个节点 / 做了什么 / payload）+ **投影出的当前状态表**；状态表可由事件流完整重建。
- **理由**：FR-13 全部审计与「还原现场」需求免费获得；D-05 租约回收与接力审查报告、D-06 `attempts[]`、FR-12.2 聚合进度都是事件的自然投影。这是从 Temporal 内部设计借来的最有价值模式（附录 A）。
- **影响**：14.1 可观测性的追踪标识 = 事件 id 链；UI 时间线直接读事件表。PG 下加分项：事件与投影同事务写保证原子一致；`NOTIFY` 挂在事件插入上，UI/Worker 实时感知图变化。

## D-18 · 认证与权限（新增）✅ 已确认

- **决策**（用户拍板 2026-07-16）：v1 **无认证、无 token、无权限系统**。Task Service 监听 localhost，单人使用，所有调用方（人 / Agent）全权可信。
- **影响**：
  - 14.1「安全与权限」三条整体**降级为非目标**（v1 不实现独立权限、最小权限、权限拒绝审计）；
  - [05-角色与协作](./05-角色与协作.md) 的人机权限矩阵**保留但换实现**：AI 不能 Close/Pause Package 不再靠鉴权拦截，而是靠**约定**——范围类 API 仅由 UI / CLI 暴露给人使用，不写进 Agent 的工具集与提示词；`manual: true` 字段保留，作为 API 层的软校验（拒绝带 agent 标识的调用方）；
  - `owner` / `actor` 字段**保留**：它们的作用是审计与租约归属（谁领的任务、谁触发的事件），不是鉴权，D-17 事件表依赖它们；
  - D-14（人类可 Claim）自然成立——没有权限系统，人和 Agent 本就走同一接口。
- **升级路径**：若未来开放局域网/多人，加一层反向代理 basic auth 即可起步，核心协议不变。

## 附录 A · 已否决备选：Temporal 作核心状态机

**否决理由**（2026-07 评估）：

1. **概念错配**：本系统要求 DAG 是第一公民、可被用户和 Replanner 运行时改写（[12](./12-难点与产品定位.md) §12.3）；Temporal 是 Workflow-as-Code，图结构隐含在代码执行历史里，套用等于亲手把 DAG 变回内部实现，且运行时改图与确定性重放模型持续冲突。
2. **执行模型不合**：本系统 Worker 是外部 AI 进程、Pull 模型（FR-05.2）；Temporal Activity 假设 worker 托管代码，外部完成模式下 Temporal 只剩租约管理职责，而租约用 SQLite 约 20 行 SQL 可解（D-05）。
3. **生态风险**：Temporal Rust SDK 截至 2026-05 仅 Public Preview，官方标注 prerelease、API 持续变动。

**保留其设计模式**：Event Sourcing → D-17；Activity heartbeat/timeout → D-05；Retry Policy + 幂等 → D-06。

**未来可进场的位置**：v2 若 Integration/Deploy 节点变成跨小时、带人审、需 saga 补偿的长流程，可让**单个 Integration Task 的内部执行**跑在 Temporal 上——那是 Workflow-as-Code 的主场，届时 Rust SDK 也应已 GA。

## 附录 B · 已否决备选：SQLite 作首版数据库

> 历史记录：整理者最初以「Cargo 式零部署分发」为由建议 SQLite（当时的论证：单二进制可跑、状态即文件、写串行化免费送 Claim 原子性）。**用户于 2026-07-16 拍板 PostgreSQL**，理由与场景修正如下，保留本附录防止翻案时丢上下文。

1. **前提修正**：SQLite 方案的核心论据是「分发给陌生用户时的零部署」。但本项目 v1 是**单人自用工具**（D-18），不存在分发问题；开发者本机已有 PG，部署成本为零，该论据不成立。
2. **PG 的实际增益**（对本项目负载而言）：
   - `FOR UPDATE SKIP LOCKED`：多 Worker 并发取任务的教科书原语，claim-next 一条语句原子完成（D-05）；
   - `LISTEN/NOTIFY`：Worker 空闲等待可从轮询升级为推送唤醒，事件插入即广播（D-17）；
   - `WITH RECURSIVE` + 部分索引：祖先链检查（D-16 派生 Blocked）、受影响子图（FR-02.4）、Ready 查询（14.1 性能 1）都有原生高效支撑；
   - JSONB：Task 契约、attempts[]、事件 payload 半结构化字段免建表。
3. **放弃的东西，且可接受**：「状态文件跟项目走」（备份=拷文件）——单人场景用 `pg_dump` 脚本即可；「测试用内存库」——用 testcontainers 或本机 PG 的临时 schema 替代。

**回到 SQLite 的触发条件**：项目转向「分发给他人使用的独立 CLI 工具」定位时重新评估（届时嵌入式零依赖重新变成第一优先级）。
