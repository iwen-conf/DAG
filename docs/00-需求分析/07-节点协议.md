# 07 · 节点协议

对话明确要求：**真正应该定义的是节点协议**。Scheduler 与 Agent 只依赖协议字段，不依赖业务语义。

## 7.1 节点字段（草案）

```yaml
id: server.identity.login.api

type: task      # task | package

state: ready

depends:
  - server.identity.contract

children: []    # package 展开后非空；task 必须为空

artifact:
  - login_api.go

owner: null

validator: compile   # 或一组校验器标识
```

### 字段说明

| 字段 | 说明 |
|------|------|
| `id` | 全局唯一，建议层级化命名 |
| `type` | `package` / `task` |
| `state` | 见 [04-状态机.md](./04-状态机.md) |
| `depends` | 前置节点 id 列表 |
| `children` | 子节点 id 列表 |
| `artifact` | 预期/实际产物清单 |
| `owner` | Claim 后的 Agent id；未领为 null |
| `validator` | 验收方式 |

### Package 扩展字段（建议）

```yaml
type: package
state: Closed
reason: 业务暂缓
manual: true          # 该状态仅人类可改
until: 2026-10-01
capability_hint: []   # 可选：对子任务能力的默认提示
```

### Task 扩展字段（建议）

```yaml
type: task
required_capabilities:
  - rust
  - database
priority: 0
```

## 7.2 调度判定伪代码

```go
if node.Type == Task &&
   node.State == Ready &&
   node.DependsSatisfied() &&
   parentPackageAllowScheduling(node) {
    Claim(node)
}
```

其中 `parentPackageAllowScheduling` 应拒绝：

- 任意祖先 Package 为 `Closed` / `Paused` / `Ignore` / `Archived`（具体集合实现时固化）。

## 7.3 类型不变量

```text
type == package  →  不可 Claim；可 Expand（状态允许时）
type == task     →  children == []；可 Claim；不可 Expand
```

## 7.4 示例：关闭 Package 后的传播

```text
Payment (Package) → Closed
  Contract (Task) → Blocked
  Application (Task) → Blocked
  Infrastructure (Task) → Blocked
  Tests (Task) → Blocked
```

Scheduler 在 `next` 查询中直接过滤这些节点。

## 7.5 协议稳定性要求（需求级）

1. 增加业务类型（Login、Payment）**不得**修改 Scheduler 代码；
2. 新 Validator 通过注册扩展，而非硬编码进调度核心；
3. 状态迁移表版本化，便于多 Agent 兼容。
