# 13 · 详细功能需求清单

> 本文把对话中的方向性描述转成可编号、可测试的产品需求。带“待确认”的条目来自对话建议，不代表用户已经最终拍板。

## FR-01 需求与架构建模

- **FR-01.1** 系统应接收自然语言目标、范围限制、技术约束和验收意图。
- **FR-01.2** 系统应保存 Requirement、Architecture、Contract、Package、Task 之间的来源关系。
- **FR-01.3** Planner 应先识别业务动作、变化原因、生命周期、事务边界和 Actor，再生成业务边界。
- **FR-01.4** 人工应能编辑、确认或拒绝 Planner 的架构与拆分结果。

## FR-02 层级可演化 DAG

- **FR-02.1** Package 可包含子 Package 或 Task，形成“大包小”的层级结构。
- **FR-02.2** 父子组织关系与 `depends` 执行依赖必须分别存储，不能混为一条边。
- **FR-02.3** 系统应支持按需展开 Package，不要求首次生成完整项目的所有叶子。
- **FR-02.4** 系统应查询节点的祖先、后代、前驱、后继及受影响子图。
- **FR-02.5** 只有叶子 Task 可以被执行；Package 永远不可被 Worker Claim。

## FR-03 Planner 与 DAG Compiler

- **FR-03.1** Planner 可把 Package 拆成更小的 Package 或符合最小标准的 Task。
- **FR-03.2** 每个 Task 草案必须包含目标、输入、输出、约束、写范围、依赖、能力和 Validator。
- **FR-03.3** 编译校验必须检测：环、悬空依赖、重复 ID、类型不变量、缺失输入和并行写冲突。
- **FR-03.4** 非法 DAG 不得进入可执行状态，并应返回具体节点、冲突资源和依赖路径。
- **FR-03.5** 验证失败、需求变化或发现新依赖时，应支持只重规划受影响子图。
- **FR-03.6** 已完成且输入版本未变化的节点应可复用，不应无条件全量重跑。

## FR-04 Task 契约

- **FR-04.1** Task 必须只有一个主要目标和一个主要输出或不可分割的产物集合。
- **FR-04.2** 输入必须引用明确的 Artifact 及版本，不能只依赖会话记忆或“工作区最新状态”。
- **FR-04.3** Task 必须声明独占写入的文件、目录、资源或逻辑所有权范围。
- **FR-04.4** Task 必须声明可独立执行的验收标准。
- **FR-04.5** 输入未冻结、写范围冲突或 Validator 不可用时，Task 不得进入 Ready。

## FR-05 Task Service 与 Pull 调度

- **FR-05.1** 所有 Worker 通过中央 Task Service 协作，不依赖 Worker 间直接通信。
- **FR-05.2** Worker 应主动 Pull 可执行任务，Scheduler 不采用 Boss 主动推送模型。
- **FR-05.3** Scheduler 只依赖节点类型、状态、依赖、能力、父级范围和写集合，不硬编码业务语义。
- **FR-05.4** `Claim` 必须是原子操作，同一 Task 不能被两个 Worker 同时成功认领。
- **FR-05.5** Scheduler 必须过滤依赖未满足、能力不匹配、写冲突或祖先 Package 禁止调度的 Task。
- **FR-05.6** 无 Ready Task 时，Worker 可继续轮询；系统可请求 Planner 下钻未展开 Package。

## FR-06 Worker 生命周期

- **FR-06.1** Worker 启动时应注册能力标签，不要求绑定固定业务身份。
- **FR-06.2** Worker 可执行 Claim → Running → 提交 Artifact → Review/Validate 的循环。
- **FR-06.3** Worker 应上报在线、忙闲、当前 Task、执行进度和错误。
- **FR-06.4** Worker 异常退出后的任务回收必须有明确机制；租约和心跳细节待确认。
- **FR-06.5** Worker 不得写入 Task 未声明的范围；越权写入应导致验证失败。

## FR-07 Artifact Store

- **FR-07.1** Worker 完成工作的主要交付物必须是 Artifact，而不是自然语言完成说明。
- **FR-07.2** Artifact 应记录内容摘要、版本、生产 Task、执行者、输入版本和产生时间。
- **FR-07.3** 后继 Task 只能消费已发布且满足所需验证条件的 Artifact。
- **FR-07.4** 系统应从 Artifact 反查生产链路，从 Task 反查原始需求和 Contract。
- **FR-07.5** Artifact 后端可扩展；首版采用文件、Git 或对象存储尚待确定。

## FR-08 Validator 与失败处理

- **FR-08.1** Task 必须绑定一个或多个可用 Validator。
- **FR-08.2** Validator 可执行编译、Lint、单元测试、API 校验、Schema 校验和未声明写入检查。
- **FR-08.3** Validator 通过后 Task 才能进入 Done。
- **FR-08.4** 失败结果必须记录根因、日志、输入版本和不满足的契约，不能静默吞错。
- **FR-08.5** 失败后采用原 Task 重试、创建修复 Task 或触发 Replanner，策略待确认。

## FR-09 Package 范围管理

- **FR-09.1** Package 应拥有独立于 Task 的管理状态机。
- **FR-09.2** 人工可 Pause、Close、Reopen 和 Archive Package。
- **FR-09.3** Close 后禁止 Claim 子 Task、继续 Expand、自动生成子任务及 Planner 修改。
- **FR-09.4** Scheduler 必须跳过被禁用 Package 的整棵子树。
- **FR-09.5** 范围操作必须记录操作者、时间、原因和可选恢复条件。
- **FR-09.6** AI 默认无权 Close、Reopen 或 Delete Package。
- **FR-09.7** `Ignore` 是否作为独立永久放弃状态，待确认。

## FR-10 Contract First 与并行开发

- **FR-10.1** 系统应支持先生成并冻结 Domain、Port、DTO、Event、OpenAPI、Schema 等 Contract。
- **FR-10.2** Business 依赖自身定义的 Port，Infrastructure 实现 Port，不得让业务依赖具体基础设施。
- **FR-10.3** Contract 冻结后，Business、Infrastructure、Controller、SDK、Mock 和客户端可按写范围并行。
- **FR-10.4** Contract 版本变化后，系统应计算并标记受影响后继节点。
- **FR-10.5** 破坏性 Contract 变更的发布权限与审批策略待确认。

## FR-11 Integration 与 Merge

- **FR-11.1** 普通并行 Task 不得共享可变写入目标。
- **FR-11.2** 多产物组合应由显式 Integration Task 完成。
- **FR-11.3** Merge 冲突必须生成明确冲突报告，不能静默选择任意一方。
- **FR-11.4** Git 工作树、分支或沙箱隔离方案待确认。

## FR-12 可视化与操作台

- **FR-12.1** 用户应能分别查看层级树和执行依赖图。
- **FR-12.2** UI 应展示 Package 聚合进度与 Task 实际执行状态。
- **FR-12.3** 节点详情应展示输入、输出、Owner、能力、Validator、错误和历史。
- **FR-12.4** 系统应可视化阻塞原因、缺失依赖、写冲突及受影响子图。
- **FR-12.5** 大型项目应支持节点懒加载和按子树操作。

## FR-13 审计与历史

- **FR-13.1** 规划、展开、认领、状态迁移、验证、重规划和人工范围操作均应留存事件。
- **FR-13.2** DAG 和 Contract 应版本化，并能查看差异和变更原因。
- **FR-13.3** 系统应能还原某个 Artifact 当时使用的图版本、输入和执行环境。
