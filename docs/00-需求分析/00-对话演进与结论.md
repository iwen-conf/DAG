## 1. 一句话共识

> **用 Contract 定边界，用层级可演化 DAG 定顺序，用 Pull + Claim 调度自治 Worker，用 Artifact 替代长会话 Memory，人只控范围（Close / Pause / Reopen）。**

产品定位不是「又一个 Agent 框架」，而是 **AI 时代的 Cargo / Bazel + 项目操作系统**：规划、调度、验证、增量重建；大模型只是执行器之一。

## 3. 从「拆边界」到「操作系统」的抬升路径

```text
上下文边界（DDD）
    ↓
小任务四条件（AI 可完成）
    ↓
并行五条件（无共享写 + DAG）
    ↓
Task IR / Planner（图从哪来）
    ↓
Contract First（冻结边界才能并行）
    ↓
层级可演化 DAG（大包小、按需展开）
    ↓
Task Service + Pull + Claim（工程运行时）
    ↓
人控范围（Close / Pause）+ 三角色（Planner / Worker / Validator）
```

## 4. 不可妥协的共识（需求级）

1. **交付物是 Artifact，不是对话。** 不要用越来越长的 Memory 扛项目。
2. **DAG 是第一公民**，不是 Agent 框架内部的隐藏实现。
3. **Contract 决定边界，DAG 决定顺序，Agent 只实现节点。**
4. **静态完整 DAG 不适合真实开发**；要初始图 + 验证 + 受影响子图重规划。
5. **只有叶子 Task 可 Claim**；Package 管范围与展开。
6. **Scheduler 不懂业务**；只懂 type / state / depends / capability。
7. **Pull 不 Boss 派发**；Claim 原子防抢。
8. **范围决策仅人类**：Pause / Close / Reopen / Delete Package。

## 5. 核心对象三元组

| 对象 | 作用 |
|------|------|
| **Contract** | 冻结接口与规则，使并行安全 |
| **DAG** | 表达依赖、顺序、可展开层次 |
| **Artifact** | 节点输入/输出的真实载体（文件/产物） |

对比常见 Agent 框架：

| 常见框架 | 本系统 |
|----------|--------|
| Conversation | Artifact |
| Memory | 节点 IO + 产物库 |
| Agent 工作流 | 工程 DAG + 状态机 |
