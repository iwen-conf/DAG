# 03 · 节点模型

## 3.1 层级 DAG（大包小）

对话共识：平铺几百个节点难管理；应升级为 **Hierarchical DAG**。

类比：

| 系统 | 层级 |
|------|------|
| Bazel | Package → Target → Action |
| Kubernetes | Namespace → Deployment → Pod |
| 编译器 | Module → Function → Basic Block |

推荐组织层级（语义标签，实现可合并相邻层）：

```text
Project（项目）
  └── Package（包 / 工作空间）
        └── Module（模块 / Context）
              └── Feature（功能）
                    └── Task（任务）← 唯一可 Claim
```

示例：

```text
Project
└── Server
    └── Identity
        └── Login
            ├── 实现 LoginService   (Task)
            ├── 编写 LoginRepository (Task)
            ├── 编写 Login API      (Task)
            └── 编写 Login Test     (Task)
```

前几层负责**组织与调度**；最后一层负责**执行**。

## 3.2 运行时二分：Package vs Task

调度与认领只需两种类型：

| 类型 | 含义 | 能否 Claim | 职责 |
|------|------|------------|------|
| **Package** | 可展开的工作空间 / 目录节点 | 否 | 范围管理、Expand、聚合子状态 |
| **Task** | 可执行叶子 | 是 | 开发、提交 Artifact、验收 |

中间的 Module/Feature 在运行时均可落为 `type: package`，仅叶子为 `task`。

### 结构示例

```text
Project
├── Server      (Package)
├── Client      (Package)
└── DevOps      (Package)

Server
├── Identity    (Package)
├── Novel       (Package)
└── User        (Package)

Identity
├── Contract        (Task 或再拆)
├── Infrastructure  (Task)
├── Application     (Task)
└── Controller      (Task)
```

## 3.3 判定规则

```text
展开后的 Package：可有 children
Task：children == []
```

补充：

- 未展开 Package 可 `Planning`，children 可空，**不可 Claim**；  
- 仅 `type == task && state == Ready` 可被 Worker 认领；  
- Scheduler **永不执行** Server 这类目录节点。

## 3.4 Lazy Expand（按需展开）

Planner / 用户 / 空闲 Worker 可触发：

```text
Expand(Server) → Identity, Novel, Comment
Expand(Identity) → Contract, Application, Infrastructure
```

- 不必一次生成整图；  
- 规划本身递归（AI 擅长分层规划，不擅长一次 500 Task）；  
- 类似 Lazy Loading。

递归过程：

```text
Package →（Planner Expand）→ Package → … → Task →（Worker）→ Artifact
```

## 3.5 Package = Workspace（管理对象）

关闭 `Payment` Package → 子树全部 Blocked，无需逐 Task 关闭。

呈现类似 GitHub Project：

```text
📦 Server        Running
  📦 Identity    Completed
  📦 Novel       Running
  📦 Payment     Closed
  📦 Search      Planning
```

状态机见 [04-状态机.md](./04-状态机.md)。

## 3.6 依赖与调度

调度只关心：

- Task 的 `depends` 是否满足；  
- 祖先 Package 是否允许调度（非 Closed/Paused/Ignore）；  
- 写权限 / capability 是否匹配。

Package 的 Running 是**聚合态**（子树有任务在跑），不是自己写代码。

## 3.7 工程角色标签（可选）

与 Package/Task 正交，节点可打标签（见 [10](./10-Contract-First与架构并行.md)）：

| 标签 | 含义 |
|------|------|
| Specification | 规范：Requirement / OpenAPI / Schema |
| Generator | 生成实现：只读规范 |
| Validator | 只判断，不产业务代码 |
| Integration | Merge / Deploy / Release |

## 3.8 节点数据结构（统一）

```yaml
id: server.identity
type: package          # package | task
status: pending
children:
  - contract
  - application
  - infrastructure
depends_on:
  - shared.openapi
artifacts:
  - identity/*
owner: null
```

更完整字段见 [07-节点协议.md](./07-节点协议.md)。

## 3.9 展开责任

| 角色 | 行为 |
|------|------|
| Planner | 递归展开 Package；失败后重规划子图 |
| Worker | 只 Claim Task；无任务时可触发/请求 Expand |
| 人类 | Close/Pause 等范围操作；可手动请求 Expand |

多 Planner（Go / Vue / DB）可并存，均不直接承担范围决策。
