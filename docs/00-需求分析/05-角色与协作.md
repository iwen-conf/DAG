# 05 · 角色与协作

## 5.1 三层角色 + 人类

| 角色 | 职责 | 不做 |
|------|------|------|
| **Planner** | 维护/演化 DAG；展开大节点；验证失败后重规划受影响子图；插入新依赖节点 | 不写业务实现代码 |
| **Worker** | Claim → 开发 → Commit → 更新状态 | 不做架构范围决策 |
| **Validator** | 确认产物是否符合 Contract（如 compile） | 不负责任务拆分 |
| **人类（Tech Lead / PM）** | 范围决策：Pause / Close / Reopen / Delete；关键架构确认 | 不需要手写每个叶子任务 |

> Tech Lead 角色「不一定是人」，也可以是特殊 Planner Agent；但 **范围控制类操作默认仅人可做**。

## 5.2 Planner 的具体工作

1. 检查 DAG 是否还有未展开的大节点；
2. 判断任务是否过大，需要继续拆分；
3. 根据验证失败重新规划受影响子图；
4. 发现新依赖后动态插入节点。

多 Planner 协作（优势）：

- Go 后端 Planner
- Vue 前端 Planner
- 数据库 Planner  

都只生成/细化 Package 与 Task，不直接生成最终业务代码（或仅生成骨架需另约定）。

## 5.3 Worker 的具体工作

```text
Loop {
  看有没有 Ready
  Claim
  做
  Commit
  Validate
}
```

- 启动时上报 Capability（Go / Vue / Rust / Database …）
- 只领取匹配能力的 Task
- Agent 之间不通信，只打 Task Service

## 5.4 人机权限矩阵（对话共识）

| 操作 | AI | 人 |
|------|----|----|
| Claim Task | ✅ | ✅ |
| 完成 Task | ✅ | ✅ |
| Expand Package | ✅（可配置） | ✅ |
| Pause Package | ❌ | ✅ |
| Close Package | ❌ | ✅ |
| Reopen Package | ❌ | ✅ |
| Delete Package | ❌ | ✅ |

含义：

- AI **不能擅自砍掉或暂停**模块；
- 影响项目范围的决策始终由人控制；
- AI 负责规划细化与执行。

建议在节点上增加 **Manual** 标记：标明哪些状态迁移禁止 AI 触发。

## 5.5 与「多 Agent 聊天」的区别

本系统模拟的是：

> 有任务板、有状态流转、有角色分工的 **工程组织**。

不是多个 AI 在同一个对话里讨论。
