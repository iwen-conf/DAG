# 06 · 调度与任务服务

## 6.1 中央 Task Service

```text
                DAG Server / Task Service
                         │
            ┌────────────┼────────────┐
            │            │            │
        Agent A      Agent B      Agent C
```

- 所有 Agent **不要互相通信**；
- 全部访问 Task Service；
- Task 的状态是协作的唯一真相源。

## 6.2 Pull Model（明确反对 Boss 派发）

### 不采用（对话评价不好）

```text
CrewAI 风格：Boss → Worker
```

Scheduler **不应该主动分配**。

### 采用

```text
Agent：有没有任务？
Scheduler：有。
Agent：认领。
结束。
```

好处：Agent 完全自治；空闲即拉取；易于水平扩展。

## 6.3 Scheduler 的「无知」原则

类比 Kubernetes：

```text
Deployment → ReplicaSet → Pod
Scheduler 不知道 Pod 里业务是什么
只知道 Pending / 哪个 Node 空闲
```

本系统：

```text
Project → Package → Module → Task
Scheduler 不知道 Login 是什么
只知道 Ready / 哪个 Agent 空闲 / 能力是否匹配
```

## 6.4 能力匹配

Agent 启动：

```text
Hello.
Capability: Go, Vue, Rust, Database  # 示例
```

Scheduler 记录：

```yaml
agent:
  go: true
  vue: false
  rust: true
```

只把 Rust Task 暴露给具备 Rust 能力的 Agent。

> 对话中甚至提到：「Agent 不应该知道自己是谁」——更强调能力标签与任务标签匹配，而非硬编码角色身份。

## 6.5 最小 API 轮廓（来自对话示例）

| 接口 | 行为 |
|------|------|
| `GET /task/next` | 返回下一个可 Claim 的 Ready Task（按 capability 过滤） |
| `POST /claim` | 原子认领；成功后 `owner` 写入，他人不可再领 |

实现约束：

- Claim 必须原子（防止双 Agent 同领）；
- Closed / Blocked / 依赖未满足的节点永不进入 next；
- 可选：按 priority、fairness 排序（对话未展开，列为后续设计）。

## 6.6 与 Validator / Artifact Store 的协作

```text
Worker Commit Artifact
  → Artifact Store
  → Validator（如 compile）
  → 通过则 Task → Done
  → 失败则 Task → Failed，并可能触发 Planner 重规划
```

具体失败策略在分享对话中未细化，需在后续设计文档中补齐。
