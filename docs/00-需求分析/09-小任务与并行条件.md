# 09 · 小任务与并行条件

> 来源：完整对话第 2–3 轮  
> 用途：定义「什么算一个可交给 AI 的 Task」以及「何时可并行」。

## 9.1 为什么长任务容易失败

上下文有容量与注意力限制。让 AI「开发整个小说网站后台」需同时维护用户/权限/DB/API/前端/评论/搜索/部署等十几个概念 → 遗忘、分心、幻觉、自相矛盾。

> 正确做法：**不是让 AI 做一个大任务，而是目标拆成多步 Task。**

## 9.2 小任务四条件（必须满足）

一个 AI 任务最好同时满足：

| 条件 | 含义 |
|------|------|
| **一个目标** | 只解决一个问题 |
| **一个输入** | 依赖少且明确 |
| **一个输出** | 一种结果（一种产物形态） |
| **一个验证标准** | 能明确回答「完成了吗？」 |

若同时含多目标、多输入、多交付物 → **继续拆**。

### Prompt 拆分经验法则

若出现：

```text
还有…… / 另外…… / 顺便…… / 最后……
```

几乎应拆成多个任务。

### 输出形态示例

```text
Task1  输入：需求        输出：schema.sql
Task2  输入：schema.sql  输出：OpenAPI.yaml
Task3  输入：OpenAPI     输出：Go Server
```

这就是流水线（Pipeline）。

### 可独立验证

```text
生成用户表 → ✓ 字段完整 ✓ 主键 ✓ 唯一索引 ✓ 外键 ✓ 能迁移
```

反例：`开发用户模块` —— 无人知道什么叫完成。

### 单一知识域

讨论 `User` 时不要突然塞入 Redis / OSS / Docker / MQ / 短信 / 邮件，否则 Context 再次污染。

## 9.3 Task Tree 形态

成熟系统常形成任务树：

```text
Goal
├── Analyze
├── Plan
├── Design
├── Implement
│   ├── User / Novel / Chapter / …
│   │   ├── Model / API / Service / Test / Docs
├── Review
├── Refactor
└── Deliver
```

最小任务可细到一句话：

> 根据已有 Schema 实现 `UserService.Create()`，不要修改其他文件。

## 9.4 并行的本质

> **并行的本质不是「任务小」，而是任务之间没有共享可变状态（Shared Mutable State）。**

### 不能并行

| 情况 | 原因 |
|------|------|
| 两 Task 都改同一 `User` 表/文件 | 写冲突 |
| 前端依赖尚未冻结的 API | 依赖未满足 |

### 能并行

```text
            Schema（已冻结）
               │
      ┌────────┴────────┐
  User API         Novel API
      │                 │
  Backend/Docs…    Backend/Docs…
```

User API 与 Novel API 无共享写 → 可多 Agent 同时写，最后 Merge。

## 9.5 可并行 AI Task 的五条件

| # | 条件 | 说明 |
|---|------|------|
| 1 | **输入固定** | 依赖输入已冻结，执行中不变 |
| 2 | **输出唯一** | 只负责一种明确产物 |
| 3 | **拥有唯一写权限** | 只有当前 Task 可写其资源，其他只读 |
| 4 | **依赖关系明确** | 形成 DAG，无环、无互相等待死锁 |
| 5 | **可独立验证** | 节点完成可单独检查 |

满足后，2 / 20 / 200 个 AI 在调度语义上等价于编译器任务调度。

## 9.6 下一代模型：不是 Context，而是 IO

许多 Agent 维护巨大 Context；对话主张：

```text
Input → Transform → Output
```

类比 UNIX：`stdin → process → stdout`。

边是 **Artifact（文件/产物）**，不是 Memory、不是聊天记录。

## 9.7 决策权（Decision Ownership）

真正该拆的还有 **写权限归属**：

| Agent | 唯一可写 | 其他 |
|-------|----------|------|
| Schema Agent | `schema.sql` | 只读 |
| API Agent | `openapi.yaml` | 只读 |
| Go Agent | `go/*` | 只读 |

→ 避免两 Agent 同改一物。

## 9.8 Task 描述最小结构

```text
Task
  Input
  Output
  Constraint   # 如：不能修改 schema
  Owner
```

示例：

```text
Task: 生成 User API
Input: schema.sql
Output: openapi/user.yaml
Constraint: 不能修改 schema
Owner: Agent-1
```

## 9.9 设计重心

> **真正需要设计的不是 Prompt，而是任务依赖图（Task DAG）与每个节点的输入/输出契约（Contract）。**
