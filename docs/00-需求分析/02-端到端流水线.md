# 02 · 端到端流水线

> 边界如何拆见 [08](./08-边界拆分方法论.md)；图如何编译见 [11](./11-Task-IR与规划器.md)；契约如何冻结见 [10](./10-Contract-First与架构并行.md)。

## 2.1 错误简化 vs 正确链路

### 不要这样

```text
DAG → Task → Agent
```

或：

```text
自然语言需求 → 直接开多个 Agent 聊天
```

直接把「图」当成「可执行任务」、或跳过架构与契约，会导致：大节点无法执行、无法并行、无法范围管理、写冲突与环依赖。

### 应该这样

```text
需求
  ↓
Architecture（架构 / Bounded Context 划分）
  ↓
Contract（端口与 API 等契约冻结）
  ↓
DAG（依赖图 / 规划，可演化、可层级展开）
  ↓
Task（可执行叶子任务）
  ↓
Scheduler（调度）
  ↓
Agent（执行）
  ↓
Artifact（产物）
  ↓
Validator（验证）
  ↓ 失败时
Replanner（只改受影响子图）
```

## 2.2 各阶段职责

| 阶段 | 输入 | 输出 | 谁负责 |
|------|------|------|--------|
| 需求 | 业务目标、约束 | 需求描述 / Requirement Artifact | 人 |
| Architecture | 需求 | Context 边界、稳定性分层 | 人 + Planner（可配置） |
| Contract | 架构 | Port / OpenAPI / Schema 等冻结契约 | Planner + 人确认 |
| Task IR / DAG | 架构 + 契约 | 可生长的层级图 + 依赖 | Planner + 规则校验器 |
| Task | 叶子节点 | 可 Claim 的执行单元 | Planner Expand 到底 |
| Scheduler | Ready Task 池 | 可被 Claim 的任务视图 | Task Service / Scheduler |
| Agent | Claimed Task | 代码/文档等 Artifact | Worker Agent |
| Validator | Artifact + Contract | 通过 / 失败 | Validator |
| Replanner | 失败节点 + 子图 | 更新后的 DAG 片段 | Planner |

## 2.3 DAG 与 Task 的关系

- **DAG 是规划**：例如 `Server → Identity / Novel / Payment`。
- **Task 是叶子**：例如 `Generate Login API`、`Generate Unit Test`。
- Identity 若仍是 Package，**不能**被 Agent Claim；必须继续展开。

## 2.4 递归展开（DAG 会「长」）

```text
Project
  └─ Server（Package，非叶子）
       └─ Expand → Identity, Novel, Comment
            └─ Identity（仍非叶子）
                 └─ Expand → Contract, Application, Infrastructure
                      └─ Application
                           └─ Expand → Login, Register, ResetPassword
                                └─ Login
                                     └─ Expand → Generate DTO / API / Tests  ← 真正的 Task
```

规则：

- **Package**：`children != []`（或尚未展开完成）
- **Task**：`children == []`，且可执行

Scheduler **永远不会**执行 `Server` 这类目录节点，只执行叶子 Task。

## 2.5 Worker 主循环（目标行为）

```text
Loop {
    看有没有 Ready
    Claim
    做
    Commit
    Validate
}
```

无 Ready 时：

- 不退出；
- 再检查任务板；
- 或下钻一层：是否有 Package 需要 Expand / 是否有依赖自己的子任务。

## 2.6 并行执行语义

当依赖满足后，互不依赖的叶子可同时 Claim，例如：

```text
Identity.Contract Done, Novel.Contract Done
  → 并行：
      Agent1 → Identity.Application
      Agent2 → Novel.Application
      Agent3 → Identity.Infrastructure
      Agent4 → Novel.Infrastructure
```
