# 01 · 系统愿景

## 1.1 问题定位

当前很多 Agent 框架把 AI 当作「执行器」或「互相聊天的角色」。本系统的目标不同：

> 设计一个 **AI Project Operating System（AI 项目操作系统）** / **AI Build System（AI 构建系统）**。

它更接近：

| 隐喻 | 吸收什么 |
|------|----------|
| **Cargo / Bazel** | 规划依赖图、调度构建、验证产物、增量重建 |
| **GitHub Projects / Jira** | 任务板、状态流转、范围管理 |
| **Kubernetes 调度器** | 只关心 Ready / Pending / 能力，不关心业务语义 |
| **真实软件团队** | Planner / Developer / Reviewer + 人工范围决策 |

而不是：

- CrewAI 式 Boss → Worker 的推送分工  
- LangGraph 式「多 Agent 对话 / 工具链」图（推理 DAG ≠ 工程 DAG）  
- 以长会话 Memory 为项目真相源的聊天 Agent  

对话中的产品一句话：

> 重点从「Prompt 怎么写」转向「任务图怎么生成、验证和调度」。

## 1.2 核心对象三元组

```text
Artifact + DAG + Contract
```

| 对象 | 作用 |
|------|------|
| **Contract** | 冻结接口与规则 → 并行安全 |
| **DAG** | 依赖、顺序、层级展开 |
| **Artifact** | 节点真实输入/输出（文件/产物） |

Agent 只是生成 Artifact 的执行器；会话不是交付物。

## 1.3 核心隐喻：Agent = Developer

```text
while (true) {
    看任务板
    有没有可以做的？
    → Claim（认领）
    → 开发
    → Commit
    → 更新状态
    → 再去看任务板
}
```

无 Ready 时 **不退出**：

1. 再检查任务板；  
2. **下钻一层**：是否有可 Expand 的 Package / 子任务。

「无任务时下钻」是系统最关键的自治行为。

## 1.4 系统形态（逻辑架构）

```text
                 Planner / Replanner
                    │
              Expand / 演化 DAG
                    │
         ┌──────────┴─────────┐
         │                    │
     Task Service       Artifact Store
         │                    │
         ├──────────┐         │
         │          │         │
      Scheduler  Validator    │
         │                    │
         ├──────┬──────┬──────┤
         │      │      │
      Agent1 Agent2 Agent3
```

Agent **不直接通信**；只访问 Task Service。  
边是 Artifact，不是聊天消息。

## 1.5 设计原则

| 原则 | 说明 |
|------|------|
| 变化原因定边界 | 不按表/巨型 Service 拆 Context（见 [08](./08-边界拆分方法论.md)） |
| 小任务四条件 | 一目标 / 一输入 / 一输出 / 一验证 |
| 并行五条件 | 输入固定、输出唯一、独占写、依赖明确、可独立验证 |
| Contract First | Port 冻结后 Business ∥ Infra ∥ 多端 |
| Task IR + 混合规划 | AI 创造图，程序查环/写冲突 |
| 可演化层级 DAG | 大包小、Lazy Expand，非一次静态完整图 |
| 自治 Pull | 非 Boss 派发；Claim 原子防抢 |
| 调度不懂业务 | 只看 type / state / depends / capability |
| 范围由人控 | Close / Pause / Reopen / Delete Package 仅人类 |
| 能力匹配 | Agent 注册 capability，只领匹配 Task |

## 1.6 成功时的体验

1. 人给出需求；Architecture + Contract 被冻结或版本化；  
2. Planner 生成初始层级 DAG，并按需递归展开；  
3. 多 Worker 并行 Claim 无写冲突的叶子 Task；  
4. Validator 对 Artifact 做编译/契约/测试检查；  
5. 失败时 Replanner 只改受影响子图；  
6. 人可 Closed 某 Package（如 Payment），整棵子树停调度；  
7. 无 Ready 时 Worker 下钻 Expand，直至 Completed 或人工暂停。
